# 0.9.0 — Resource-Diff Plan Model + DAG Executor + Apply Artifact Emission

> **For agentic workers:** Use superpowers:subagent-driven-development.

**Goal:** Replace the ad-hoc `PlannedAction[]` model with a resource-diff structure (`DesiredState` vs `ObservedState` → `StateDiff`), convert the executor from abort-on-failure to a DAG with per-node status, and ship `apply --output` / `--emit` so `flaker apply` finally absorbs `flaker ops daily` fully. Together these three changes turn apply from a thin sequencer of imperative steps into a genuine reconciliation loop.

**Architecture:**

```
flaker.toml (DesiredState) ──┐
                             ├── computeStateDiff() ──> StateDiff ──> planApply() ──> PlannedAction[]
DuckDB / probe (ObservedState) ──┘

PlannedAction[] + ExecutorDeps ──> executePlan() ──> ExecutionResult
                                   (DAG: independent actions run concurrently,
                                    per-node status, no global abort)
```

After 0.9.0:
- Plan output names the specific state fields that drift (not just "action X because reason").
- Executor shows which actions succeeded/failed independently.
- `flaker apply --output daily` produces the same artifact as `flaker ops daily`, letting us re-deprecate `ops daily`.

**Tech Stack:** Node.js 24+, TypeScript, Commander, Vitest.

---

## Scope & non-goals

**In scope:**
- `DesiredState` / `ObservedState` / `StateDiff` types and `computeStateDiff()`
- `planApply()` refactor: takes `StateDiff` (or the raw inputs, internally computes diff) and produces `PlannedAction[]`
- Enrich `PlannedAction` with structured diff references (e.g. `collect_ci` action carries `{ matched_commits: { actual: 18, desired: 20 } }`)
- Executor DAG: define `dependsOn` edges between actions; independent actions run in parallel
- Per-node status in `ExecutionResult.executed[*].status = "ok" | "failed" | "skipped"`; only downstream of a failed action is skipped, peers still run
- `flaker plan --output <file>` writes the plan to a JSON file (basis for `--plan-file` in 1.0.0)
- `flaker apply --output <file>` writes the execution result
- `flaker apply --emit daily|weekly|incident` produces the matching `ops` artifact alongside normal apply output
- Re-deprecate `flaker ops daily` (now fully absorbed by `apply --emit daily`); keep `ops weekly` / `ops incident` as Advanced because those still need operator-style narrative artifacts
- Update `flaker status` to highlight diff fields by name (drift section already exists; extend so each unmet threshold shows actual/desired in the same style as the new plan output)
- Empirical evaluation (Iter 8) + hold-out

**Out of scope (deferred to 1.0.0):**
- `flaker apply --target <kind>` / `--target <name>` (partial reconciliation)
- `flaker apply --refresh-only` (re-probe without mutating state)
- `flaker apply --plan-file <file>` (apply a stored plan; plan↔apply handshake)
- Removing `flaker ops daily` outright (re-deprecated in 0.9.0, removed in 1.0.0)
- Cross-resource references (e.g. `flaky_tag` depends on `quarantine_manifest`)
- Event streaming / progress callbacks
- Full `kubectl diff`-style YAML diff rendering
