# 0.10.0 — Terraform-Parity Apply Surface + Docs Sweep

> **For agentic workers:** Use superpowers:subagent-driven-development.

**Goal:** Ship the remaining terraform-parity surface on `flaker apply` (`--target`, `--refresh-only`, `--plan-file`, full `--emit incident`), break-remove `flaker ops daily` as promised in 0.9.0, and sweep the documentation gaps carried over from Phase 2 / 0.8.0 / 0.9.1. Replaces what was tentatively scheduled as 1.0.0; we remain on the 0.x line until empirical stability justifies 1.0.

**Architecture:**

- Apply gains three filter/inspection flags that don't change the resource model, only what-and-when of reconciliation.
- `ops daily` is physically deleted (deprecation cycle 0.9.0 → 0.10.0 is complete).
- Docs sweep: `docs/how-to-use.md` (English) mirrors the JA cleanup; `flaker explain` gets a full umbrella section for 5 topics; `dev` moves to an Advanced subsection.

**Tech Stack:** Node.js 24+, TypeScript, Commander, Vitest.

## Scope & non-goals

**In scope:**
- `flaker apply --target <kind>` — filter the planned actions to a single action kind (e.g. `--target collect_ci` runs only that action, skips the rest).
- `flaker apply --target <name>` — fuzzy alias; same flag accepting kind or resource name. (Simplest: single `--target` flag taking the kind string.)
- `flaker apply --refresh-only` — run the probe + state-diff computation, write a plan artifact if `--output` is set, but **do not execute** any action. Effectively `flaker plan` with observed state persisted.
- `flaker apply --plan-file <file>` — load a previously-saved `PlanArtifact` (from `flaker plan --output`), skip the planner, execute the stored actions. Includes a sanity check: if the repo's current observed state diverges too far from the plan-file's `diff.drifts`, warn and require `--force`.
- `flaker apply --emit incident --incident-run <id>` / `--incident-suite <s> --incident-test <t>` — full implementation replacing the 0.9.0 stub.
- Remove `flaker ops daily` registration (and the small deprecation infra around it if no other user remains).
- Docs sweep (combining the 0.9.1 deferred items):
  - `docs/how-to-use.md` (English) parallel sweep — same 5 section-pruning as 0.9.1 JA.
  - Full `flaker explain` umbrella section in both JA and EN `how-to-use` with all 5 topics (reason/insights/cluster/bundle/context) individually documented. `cluster` already has prose; the other 4 need short synopsis + flag reference.
  - `dev` subtree relocation to a single Advanced/Maintainer subsection in `how-to-use.*.md`.

**Out of scope:**
- `1.0.0` stability commitment — deferred indefinitely. We stay on 0.x.
- Rewriting the resource-diff model (0.9.0's `StateDiff` is stable).
- Adding new `explain` topics beyond the existing 5.
- Removing `ops weekly` / `ops incident` (they carry operator narrative; apply doesn't absorb them yet).
- Deleting the `src/cli/deprecation.ts` helper (still used by `ops daily` until Task 5 removes it; maybe prune after).

### Risk notes
- **T3 (--plan-file) drift detection** is novel. The threshold "any drift kind present now that was not in the plan file" is conservative (any new drift aborts). If this proves too strict in practice (common case: quarantine_pending fluctuates), relax to "a drift kind was resolved without our knowledge" in 0.11.0.
- **T5 (ops daily removal)** is breaking. Cron setups that didn't migrate during the 0.9.0 → 0.10.0 window will break. CHANGELOG must be prominent. The 0.9.0 deprecation cycle was exactly one minor version, matching the 0.7.0 → 0.8.0 precedent.
- **T6-8 (docs)** are zero-code. If T6 finds the EN doc has additional drift not present in JA, record those findings but don't fix them in 0.10.0 — scope to the mirror of the 0.9.1 pruning.
- **`--target` single-value semantics** (T1) is simplest. Multi-target (`--target collect_ci,calibrate`) is natural follow-up; defer to 0.11.0 if demand surfaces.
