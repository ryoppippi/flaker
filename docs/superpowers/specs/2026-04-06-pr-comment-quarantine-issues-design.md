# PR Comment + Quarantine Issue Auto-Creation

## Problem

1. flaker's CI results are only visible in terminal output. Developers must dig into CI logs to see sampling decisions and test results. Competing tools (Trunk, BuildPulse) post results directly on PRs.

2. Quarantined tests have no accountability mechanism. When a test is quarantined, nobody is notified and no tracking issue exists. The `trackingIssue` field in quarantine-manifest is metadata-only.

## Design

### Feature 1: PR Comment Format (`report summarize --format pr-comment`)

Add `"pr-comment"` as a new format option to `flaker report summarize`.

#### Output format

Compact markdown optimized for GitHub PR comments:

```markdown
## flaker: Test Report

| Metric | Value |
|--------|-------|
| Profile | ci |
| Strategy | hybrid (adaptive: FNR 1.2%, divergence 0.5%) |
| Tests run | 137 / 457 (30%) |
| Estimated saved | 320 tests, ~1.8 min |

### Results

| Status | Count |
|--------|-------|
| Passed | 135 |
| Failed | 1 |
| Flaky | 1 |
| Skipped | 0 |

### New Failures

| Test | Suite |
|------|-------|
| handles timeout | tests/api.test.ts |

### New Flaky

| Test | Suite | Flaky Rate |
|------|-------|-----------|
| concurrent write | tests/db.test.ts | 12.5% |

---
*Generated by [flaker](https://github.com/mizchi/flaker)*
```

Sections "New Failures" and "New Flaky" are omitted if empty. The output includes only actionable information ‚Äî no verbose test lists.

#### Implementation

Add a `formatPrComment` function in `src/cli/commands/report.ts` that takes:
- `NormalizedReportSummary` (existing type from report)
- Optional `SamplingSummary` (from sample/run command)
- Optional `AdaptivePercentageResult` (from profile)

The `report summarize` command already accepts `--format json | markdown`. Add `pr-comment` as a third option.

#### CI Usage

```yaml
- name: Run flaker
  run: flaker run --profile ci

- name: Post PR comment
  if: github.event_name == 'pull_request'
  run: |
    flaker report summarize --format pr-comment > /tmp/flaker-comment.md
    gh pr comment ${{ github.event.pull_request.number }} --body-file /tmp/flaker-comment.md
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### Feature 2: Quarantine ‚Üí GitHub Issue (`quarantine --auto --create-issues`)

Add `--create-issues` flag to `flaker quarantine --auto`.

#### Flow

1. `quarantine --auto` detects tests exceeding `flaky_rate_threshold` with `min_runs` data
2. For each newly quarantined test:
   a. Check quarantine-manifest ‚Äî if `trackingIssue` is already set, skip
   b. Run `gh issue create` as a subprocess with title, body, and labels
   c. Write the returned issue URL back to quarantine-manifest's `trackingIssue` field
3. Print summary: "Created N issues for newly quarantined tests"

#### Issue content

Title: `[flaker] Quarantined: {suite} > {testName}`

Body:
```markdown
## Quarantined Test

| Metric | Value |
|--------|-------|
| Suite | {suite} |
| Test | {testName} |
| Flaky rate | {flakyRate}% |
| Total runs | {totalRuns} |
| Quarantine reason | auto (flaky_rate > {threshold}) |

This test was automatically quarantined by [flaker](https://github.com/mizchi/flaker) because its flaky rate exceeded the configured threshold.

### To fix

1. Investigate the root cause using `flaker reason "{suite}:{testName}"`
2. Fix the test
3. Remove from quarantine: `flaker quarantine --remove "{suite}:{testName}"`
```

Labels: `["flaky-test", "quarantine"]`

#### GitHub API delegation

flaker does NOT call the GitHub API directly. It shells out to `gh issue create`:

```bash
gh issue create \
  --title "[flaker] Quarantined: suite > testName" \
  --body-file /tmp/flaker-issue-body.md \
  --label "flaky-test,quarantine"
```

This keeps flaker's responsibility narrow (test selection tool, not a GitHub client) and reuses the user's existing `gh` auth.

The `gh issue create` command outputs the issue URL to stdout. flaker captures this and writes it to the quarantine-manifest.

#### Error handling

- If `gh` is not installed, print a warning and skip issue creation (don't fail the quarantine operation)
- If `gh issue create` fails (auth, permissions), log the error and continue with remaining tests
- If labels don't exist in the repo, `gh` creates them automatically

## Implementation Scope

### Files to modify

- `src/cli/commands/report.ts` ‚Äî Add `formatPrComment` function and `"pr-comment"` format option
- `src/cli/main.ts` ‚Äî Wire `pr-comment` format in the `report summarize` command's format validation
- `src/cli/commands/quarantine.ts` ‚Äî Add `--create-issues` flag, `createQuarantineIssue` function using `gh` subprocess
- `src/cli/quarantine-manifest.ts` ‚Äî Add helper to update `trackingIssue` field in manifest

### New files

- `src/cli/gh.ts` ‚Äî Thin wrapper for `gh` CLI subprocess calls (issue create, check availability)

### Files unchanged

- MoonBit core ‚Äî no changes
- Storage/DB ‚Äî no changes
- Profile system ‚Äî no changes

## Testing Strategy

- Unit test for `formatPrComment`: given a NormalizedReportSummary, verify markdown output contains expected sections, omits empty sections
- Unit test for issue body generation: given quarantine data, verify markdown template
- Unit test for `gh` availability check (mock `execSync`)
- Integration: `flaker report summarize --format pr-comment` produces valid markdown
- No integration test for actual `gh issue create` (requires GitHub auth) ‚Äî tested manually
